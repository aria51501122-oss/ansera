## 第1章 はじめに

### このマニュアルについて

本マニュアルは、Ansera（アンセラ）を初めて導入されるお客様向けの**セットアップ手順書**です。

Anseraは、お客様のPC内で完結する**完全ローカルRAG（社内文書AI検索）システム**です。PDFを投入し、日本語で質問すると、根拠となる出典付きで回答が返ってきます。データは一切外部に送信されません。

本マニュアルは、PCにあまり詳しくない方でも**手順通りに進めれば導入できる**ことを目的に書かれています。専門用語はできる限り避け、必要に応じて簡単な説明を添えています。

### サポート範囲（重要）

> ⚠️ **重要: サポート範囲のご確認**
>
> 本マニュアルおよび有償サポートは、Anseraの**初回セットアップに限定**したものです。
> 継続的な月額保守、カスタマイズ開発、ワークフロー追加などは**現時点では提供しておりません**。

サポート対象範囲:

| 項目 | 対象 |
|---|---|
| 初回セットアップ作業 | ✅ 対象 |
| 動作確認・初期チューニング | ✅ 対象 |
| 動作不具合の切り分け | ✅ 対象 |
| 月額継続保守 | ❌ 対象外 |
| カスタムワークフロー開発 | ❌ 対象外 |
| n8n / PostgreSQL / Ollama の機能追加開発 | ❌ 対象外 |
| お客様独自の業務システムとの連携開発 | ❌ 対象外 |

サポート期限: **初回セットアップ完了後、30日間**（不具合の切り分け対応のみ）

## 第2章 動作環境

### 必須要件（サポート対象）

以下の要件をすべて満たすPCのみが、本マニュアルおよび有償サポートの対象です。

| 項目 | 必須要件 |
|---|---|
| OS | **Windows 11 Pro または Home（64bit）** |
| CPU | 64bit対応（Intel / AMD どちらでも可） |
| RAM | **16GB 以上** |
| 空き容量 | **20GB 以上**（SSD推奨） |
| GPU | **NVIDIA製・VRAM 8GB 以上**（GTX 1070 / RTX 3060 等） |
| ネット接続 | **初回セットアップ時のみ必要**（モデルDL約8GB） |
| 権限 | **Administrator（管理者）権限** |

### 推奨要件（より快適に動作）

| 項目 | 推奨要件 |
|---|---|
| RAM | 32GB |
| 空き容量 | 30GB（成長余地） |
| VRAM | 12GB 以上（RTX 3060 12GB / RTX 4070 等） |

### 動作不可環境（購入NG）

以下の環境では Ansera は動作しません。**ご購入前に必ずご確認ください。**

❌ **動作しません**:
- Windows 10（バージョンを問わず）
- Windows 11 32bit版
- macOS（M1 / M2 / M3 含む）
- Linux（Ubuntu / CentOS 等）
- スマートフォン / タブレット
- AMD製GPU（CUDAに非対応）
- Intel内蔵GPU
- VRAM 8GB 未満のNVIDIA GPU
- WSL2 が有効化できないPC

> ⚠️ **重要なお知らせ**
>
> Ansera パッケージに含まれる `README.md` および `setup.bat` は、技術的には Windows 10（1903以降）でも警告のみで動作を許可する作りになっています。
>
> しかし、**当マニュアルおよび有償サポートの対象は、上記「必須要件」を満たす環境のみ**です。要件を満たさない環境での動作不具合・性能不足については、サポートおよび返金の対象外となりますのでご了承ください。

### 動作環境の確認方法

[SCREENSHOT_SS01] PC環境（Windows 設定画面）

1. キーボードで `Windows キー + I` を押し、設定画面を開く
2. **システム → バージョン情報** をクリック
3. 以下を確認
   - **エディション**: 「Windows 11 Pro」または「Windows 11 Home」
   - **システムの種類**: 「64ビット オペレーティング システム」
   - **実装RAM**: 16.0 GB 以上
4. GPU確認: スタートメニュー検索で `dxdiag` → ディスプレイ タブ → 「メモリ (VRAM)」が 8192MB 以上

- [ ] RAM が 16GB 以上ある
- [ ] 空き容量が 20GB 以上ある

- [ ] 自分が管理者権限のアカウントでログインしている
- [ ] 動作不可環境に該当しないことを確認した

## 第3章 事前準備

### ⚠️ 最重要: Windowsの環境制約

> ⚠️ **重要: 以下の環境では Ansera は正常に動作しません**
>
> - **Windowsユーザー名に日本語が含まれる場合**
>   （例: `C:\Users\田中太郎\` のような環境）
> - **プロジェクトを OneDrive 配下に配置した場合**
>   （例: `C:\Users\xxx\OneDrive\Ansera\` は NG）
> - **パスに半角スペースまたは日本語が含まれる場合**
>   （例: `C:\My Documents\アンセラ\` は NG）
> - **パスの長さが 260 文字を超える場合**
>
> 該当する場合は、ドライブ直下に**半角英数のみ**のフォルダを作成し、そこに Ansera を配置してください。
>
> ✅ 推奨配置例: `C:\Ansera\`
> ✅ 推奨配置例: `D:\work\ansera\`
> ❌ NG 例: `C:\Users\山田\OneDrive\My Project\Ansera\`

ご自分のユーザー名を確認するには、コマンドプロンプトで以下を実行してください。

```
echo %USERNAME%
```

日本語が表示された場合は、新しいローカルアカウントを半角英数で作成し、そちらでログインし直してから作業を進めてください。

### Docker Desktop のインストール

[SCREENSHOT_SS02] Docker Desktop ダウンロードページ

1. ブラウザで以下にアクセス
   ```
   https://docs.docker.com/desktop/install/windows-install/
   ```
2. **Docker Desktop for Windows** のダウンロードボタンをクリック
3. ダウンロードした `Docker Desktop Installer.exe` をダブルクリック

[SCREENSHOT_SS03] Docker Desktop インストーラ画面

4. 以下のオプションに**チェックを入れる**
   - ✅ Use WSL 2 instead of Hyper-V (recommended)
   - ✅ Add shortcut to desktop
5. **OK** をクリックしてインストール開始
6. インストール完了後、**Close and restart** をクリックしてPCを再起動

### Docker Desktop の起動確認

[SCREENSHOT_SS04] Docker Desktop 起動完了画面

1. PC再起動後、デスクトップの **Docker Desktop** アイコンをダブルクリック
2. 起動には30秒〜1分程度かかります
3. 画面右下のクジラのアイコンが**緑色**になれば起動完了

> 💡 **困った時はこちら**
> 起動しない場合は第10章「Docker Desktop が起動しない」を参照してください。

### WSL2 の確認

コマンドプロンプトを開き、以下を実行します。

```
wsl --status
```

「既定のバージョン: 2」と表示されれば OK です。表示されない場合は以下を実行してください。

```
wsl --set-default-version 2
```

### 仮想化の有効化確認

`Ctrl + Shift + Esc` でタスクマネージャーを開き、**パフォーマンス** タブの **CPU** をクリック。右下の「**仮想化: 有効**」を確認します。

「無効」の場合は、PCのBIOS設定で仮想化機能（Intel VT-x / AMD-V）を有効化する必要があります。詳細は第10章「仮想化BIOS無効」を参照してください。

- [ ] Docker Desktop が起動し、緑のクジラアイコンが表示されている

## 第5章 setup.bat の実行

### 「管理者として実行」必須

[SCREENSHOT_SS07] 「管理者として実行」右クリックメニュー

1. エクスプローラで `docker\setup.bat` を開く
2. `setup.bat` を**右クリック**
3. メニューから **管理者として実行** を選択
4. 「このアプリがデバイスに変更を加えることを許可しますか？」と聞かれたら **はい** をクリック

> ⚠️ **必ず「管理者として実行」してください**
>
> ダブルクリックで起動すると管理者権限が付与されず、6段階チェックの「管理者権限」項目で停止します。

[SCREENSHOT_SS08] setup.bat 起動直後

### 6段階チェックの解説

setup.bat は起動後、以下の6項目を自動チェックします。それぞれの意味と、エラーになった場合の対処を理解しておきましょう。

[SCREENSHOT_SS09] 6段階チェック完了

| # | チェック項目 | 内容 | エラー時の対処 |
|---|---|---|---|
| 1/6 | OS バージョン | Windows 10/11 か | Windows 11 にアップグレード |
| 2/6 | 管理者権限 | 管理者として実行されているか | 一度終了し、右クリックで「管理者として実行」 |
| 3/6 | Docker Desktop | インストール済み・起動中か | Docker Desktop を起動してから再実行 |
| 4/6 | 空きディスク容量 | C ドライブに 20GB 以上 | 不要ファイル削除・別ドライブへの配置 |
| 5/6 | RAM | 16GB 以上 | メモリ増設（または別PCの利用） |
| 6/6 | NVIDIA GPU / VRAM | 8GB 以上 | サポート対象外（第2章参照） |

> 💡 **「警告」と「エラー」の違い**
>
> - **エラー**: 必須要件を満たさない場合。setup.bat は中断されます
> - **警告**: 推奨要件を満たさない場合。Y/N を聞かれ、Y で続行できます
>
> サポート対象は必須要件を満たす環境のみです。警告が出た場合は購入をお控えください。

### ランダム認証情報の自動生成

[SCREENSHOT_SS10] ランダム値生成完了

6段階チェックを通過すると、setup.bat は自動的に以下の3つの認証情報を**ランダム生成**し、`.env` ファイルに保存します。

| 項目 | 用途 | 文字数 |
|---|---|---|
| `POSTGRES_PASSWORD` | データベースのパスワード | 32文字 |
| `N8N_ENCRYPTION_KEY` | n8n の内部暗号化キー | 32文字 |
| `WEBHOOK_API_KEY` | API認証キー | 32文字 |

これらは**お客様ごとに完全にユニーク**で、外部には一切送信されません。

> ⚠️ **重要: .env ファイルは絶対に編集・削除・共有しないでください**
>
> - `.env` を編集すると、データベースに接続できなくなります
> - `.env` を削除すると、既存データが復号できなくなります
> - `.env` を他人に共有すると、お客様のPC内のデータが復号可能になります
>
> 万一の場合に備え、**`.env` は別の場所に安全にバックアップ**しておくことを強く推奨します（USBメモリの金庫保管など）。

### 自動進行する後続ステップ

setup.bat はこの後、自動的に以下を実行します（時間がかかります）。

1. Docker サービスの起動
2. PostgreSQL の起動待機（最大120秒）
3. n8n の起動待機（最大180秒）
4. Ollama の起動待機（最大120秒）
5. AIモデル `qwen3:8b` のダウンロード（**約5GB・数分〜数十分**）
6. 埋め込みモデル `bge-m3` のダウンロード（約1.2GB）
7. Postgres Credential 設定の案内表示で**一時停止**

ここで一時停止したら、第7章の手順に従って n8n の初期設定を行ってください。

### PostgreSQL Credential の作成

[SCREENSHOT_SS15] credential 作成画面

1. 左メニュー（または右上）の **Settings** をクリック
2. **Credentials** タブを選択
3. **Add Credential**（または **New Credential**）をクリック
4. 検索ボックスに `Postgres` と入力
5. **Postgres** を選択
6. 以下の値を入力

| 項目 | 値 |
|---|---|

| Host | `postgres` |

#### .env からパスワードをコピーする方法

1. エクスプローラで `docker` フォルダを開く
2. アドレスバーに `cmd` と入力して Enter（コマンドプロンプトが開く）
3. 以下を実行
   ```
   type .env
   ```
4. `POSTGRES_PASSWORD=` の右側の文字列を選択しコピー
5. n8n の Password 欄に貼り付け

## 第9章 日常運用

### バックアップ手順

Ansera は `scripts\backup.ps1` で**ワンコマンドバックアップ**ができます。

#### 標準バックアップ

1. `docker` フォルダを開く
2. アドレスバーに `powershell` と入力して Enter
3. 以下を実行
   ```
   .\scripts\backup.ps1
   ```
4. `backups\` フォルダ配下に以下が出力されます
   - `ansera-db-<日時>.sql`（PostgreSQLデータ）
   - `ansera-n8n-<日時>.tar`（n8nワークフロー・履歴）

#### 別ドライブへのバックアップ

```
.\scripts\backup.ps1 -Destination D:\Ansera\Backups
```

> ⚠️ **重要: バックアップは別ドライブまたはクラウドへ**
>
> 同じディスクにバックアップを置くと、ディスク故障時に同時消失します。
> 外付けHDD・別ドライブ・社内NAS等に転送してください。

### PowerShell実行ポリシーの対処

`backup.ps1` 初回実行時、以下のエラーが出ることがあります。

```
このシステムではスクリプトの実行が無効になっているため...
```

この場合、PowerShell を**管理者として実行**し、以下を実行してください。

```powershell
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
```

確認プロンプトで `Y` を入力します。これは**現在のユーザーのみ**に適用される設定で、システム全体には影響しません。

### 推奨: 自動バックアップの設定

Windows のタスクスケジューラで、毎日深夜に自動バックアップを実行できます。詳細手順は `README.md` の「🛠 運用ガイド → バックアップの自動化」を参照してください。

### モデルの更新

Ollamaモデル（qwen3 / bge-m3）に新版が出た場合の更新手順です。

```
docker exec ansera-ollama ollama pull qwen3:8b
docker exec ansera-ollama ollama pull bge-m3
```

差分ダウンロードのため、変化がない部分は再取得されません。

> ⚠️ **大規模バージョン跨ぎの注意**
>
> Qwen2 → Qwen3 のような世代交代では、埋め込みベクトルの次元数が変わることがあります。この場合、既存PDFの再投入が必要です。

### 月1回の定期再起動（推奨）

長期稼働で内部キャッシュが膨らむのを防ぐため、月1回の再起動を推奨します。

```
docker compose restart
```

PC再起動時にコンテナは自動復帰するため、PC再起動でも代用できます。

### 古いログの削除

アクセスログは PostgreSQL 内に保存されます。30日以上前を削除する例:

```
docker exec -it ansera-db psql -U ansera -d ansera -c "DELETE FROM access_logs WHERE timestamp < NOW() - INTERVAL '30 days';"
```

### 10.1 インストール時のトラブル

#### Docker Desktop が起動しない

- **症状**: Docker Desktop アイコンをクリックしても、クジラが緑にならない / すぐ落ちる
- **原因**: WSL2 未有効化 / 仮想化BIOS無効 / 古いバージョン
- **対処法**:
  1. PCを再起動して再度起動を試す
  2. コマンドプロンプトで `wsl --status` を実行し WSL2 が有効か確認
  3. タスクマネージャー → パフォーマンス → CPU で「仮想化: 有効」を確認
  4. Docker Desktop を一度アンインストールし、最新版を再インストール

#### WSL2 が有効化できない

- **症状**: `wsl --status` 実行で「WSLが見つかりません」と表示
- **原因**: Windows機能の「仮想マシンプラットフォーム」「Linux 用 Windows サブシステム」が無効
- **対処法**:
  1. PowerShell を管理者として実行
  2. 以下を実行
     ```powershell
     dism.exe /online /enable-feature /featurename:Microsoft-Windows-Subsystem-Linux /all /norestart
     dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart
     wsl --set-default-version 2
     ```
  3. PCを再起動

#### 仮想化BIOS無効

- **症状**: タスクマネージャーで「仮想化: 無効」と表示
- **原因**: PCのBIOSで仮想化機能が無効
- **対処法**:
  1. PCを再起動し、起動時に F2 / F10 / Del キー（PCにより異なる）でBIOS画面に入る
  2. **Advanced** または **CPU Configuration** で以下を **Enabled** に設定
     - Intel CPU: **Intel Virtualization Technology (VT-x)** + **VT-d**
     - AMD CPU: **SVM Mode** または **AMD-V**
  3. F10 で保存して再起動

#### 管理者権限エラー

- **症状**: setup.bat 実行直後に「管理者権限がありません」と表示
- **原因**: 通常のダブルクリックで起動した
- **対処法**: setup.bat を**右クリック → 管理者として実行**

#### SmartScreen ブロック

- **症状**: setup.bat 実行時に「Windows によって PC が保護されました」と表示
- **原因**: Microsoft 検証署名のないバッチファイルとしてブロック
- **対処法**:
  1. メッセージ画面の「**詳細情報**」をクリック
  2. 表示された「**実行**」ボタンをクリック

#### ウイルスバスター/Defender 干渉

- **症状**: setup.bat 実行中に処理が異常終了 / ファイル削除アラート
- **原因**: セキュリティソフトが Docker / バッチファイルを誤検出
- **対処法**:
  1. Windows Defender の場合: **設定 → ウイルスと脅威の防止 → 除外 → 除外の追加** で `C:\Ansera\` フォルダを除外
  2. ウイルスバスター等の場合: 該当ソフトのマニュアルに従いフォルダを除外
  3. 設定後 setup.bat を再実行

#### 日本語ユーザー名問題

- **症状**: パス展開エラー / Docker mount エラー
- **原因**: Windows ユーザー名に日本語が含まれる
- **対処法**:
  1. **設定 → アカウント → 家族とその他のユーザー** で半角英数の新規ローカルアカウントを作成
  2. 新規アカウントに管理者権限を付与
  3. そのアカウントでログインし直して作業を進める

#### OneDrive配下問題

- **症状**: ファイルロック / 同期コンフリクト / setup.bat が無限ループ
- **原因**: OneDrive がリアルタイム同期で Docker のファイル操作と競合
- **対処法**: `C:\Ansera\` 等のローカル直下に Ansera を移動

#### パス長260文字超

- **症状**: 「ファイル名が長すぎます」エラー
- **原因**: Windows の MAX_PATH 制限（260文字）
- **対処法**: ドライブ直下に短いフォルダ名で配置（例: `C:\a\` のような最短パス）

#### PowerShell 実行ポリシー

- **症状**: `.ps1` 実行時に「このシステムではスクリプトの実行が無効」エラー
- **原因**: Windows のセキュリティ既定設定
- **対処法**:
  ```powershell
  Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
  ```

### 10.3 docker compose 時のトラブル

#### ポート競合（5432 / 5678 / 11434）

- **症状**: `docker compose up` で「port is already allocated」エラー
- **原因**: 別アプリが同ポートを使用中
- **対処法**:
  1. 競合プロセスを特定
     ```
     netstat -ano | findstr :5678
     ```
  2. 表示された PID をタスクマネージャーで終了、または該当アプリ（PostgreSQL の別インスタンス等）を停止
  3. 解決しない場合は `docker-compose.yml` のポートマッピングを変更（例: `"15678:5678"`）

#### イメージ pull 失敗

- **症状**: `Error response from daemon: pull access denied` 等
- **原因**: ネット接続不安定 / Docker Hub 一時障害 / プロキシ環境
- **対処法**:
  1. ネット接続を確認
  2. しばらく待って再実行
  3. 社内プロキシ環境の場合は Docker Desktop の **Settings → Resources → Proxies** でプロキシを設定

#### Ollama モデル DL タイムアウト

- **症状**: `qwen3:8b` ダウンロードが途中で停止 / 異常終了
- **原因**: 回線が遅い / Wi-Fi 切断 / ディスク I/O 詰まり
- **対処法**:
  1. 有線LAN接続に切り替え
  2. setup.bat を再実行（途中までの DL は再開されます）
  3. 解決しない場合: 個別に `docker exec ansera-ollama ollama pull qwen3:8b` を手動実行

#### メモリ不足

- **症状**: コンテナが OOM（Out Of Memory）で落ちる / `docker logs` に「killed」表示
- **原因**: RAM 16GB 未満 / 他アプリが多量にメモリ使用中
- **対処法**:
  1. ブラウザ・動画ソフト等を閉じる
  2. Docker Desktop の **Settings → Resources** で割り当てメモリを増やす（最低 8GB）
  3. RAM 16GB 未満の場合は増設を検討

#### ディスク容量不足

- **症状**: コンテナ起動失敗 / 「no space left on device」エラー
- **原因**: C ドライブの空き容量不足
- **対処法**:
  1. 不要ファイル削除（ダウンロード/ ごみ箱 / Tempフォルダ）
  2. Docker Desktop の **Settings → Resources → Disk image location** を別ドライブに移動

### 10.4 n8n 設定時のトラブル

#### credential 接続テスト失敗

- **症状**: PostgreSQL credential 保存時に「Connection failed」表示
- **原因**: パスワード貼り付けミス / Host 値違い / コンテナ未起動
- **対処法**:
  1. Host が `postgres`（`localhost` ではなく）であることを確認
  2. `.env` の `POSTGRES_PASSWORD` を**前後の空白を含めずに**貼り付け
  3. Port が `5432`、Database が `ansera`、User が `ansera`、SSL が `Disable` であることを確認
  4. `docker logs ansera-db` で PostgreSQL が起動しているか確認

#### WF インポート失敗

- **症状**: setup.bat の `[7/7]` で「ワークフローのインポートに失敗」エラー
- **原因**: n8n コンテナ未起動 / `n8n-workflows` フォルダ破損
- **対処法**:
  1. `docker logs ansera-n8n` で n8n の状態を確認
  2. `docker compose restart n8n` で再起動
  3. `setup.bat --force` で再インポート

#### WF 有効化エラー

- **症状**: ワークフローが Active にならない / 灰色のままトグル不可
- **原因**: credential 紐付けエラー / WF ノード設定不整合 / alwaysOutputData 未設定
- **対処法**:
  1. 各ワークフローを開き、ノードの credential が「Postgres account」に紐付いているか確認
  2. 出力が空配列になるノードは右クリック → **Settings → Always Output Data** を ON
  3. それでも有効化できない場合: `setup.bat --force` で再インポート

#### qwen3:8b が想定外の挙動を示す

- **症状**: 回答に余計な「思考プロセス」が混入 / 出典JSONが返らない
- **原因**: Qwen3 の `<think>` モードが ON / プロンプト末尾の SOURCES_JSON reminder 不足
- **対処法**: ワークフロー内 LLM ノードのオプションで `think: false` を指定し、プロンプト末尾に出典JSON出力の reminder を追加（運用者向け補足。サポート対応事項です）

#### ブラウザキャッシュで CDN 問題に気づきにくい

- **症状**: UI の表示が古い / 修正が反映されない
- **原因**: ブラウザキャッシュ
- **対処法**: `Ctrl + F5` で強制リロード（Shift + Reload も可）
